SHA-256 has six structural weaknesses

SHA-256 hasn't been "broken" — no one has found collisions. But its architecture has known structural weaknesses that modern hash functions were specifically designed to eliminate.

01

Length extension attacks

Exploitable

If you know SHA-256(message) and the message length, you can compute SHA-256(message + attacker_data) without knowing the original message.

Secret msg H(msg) Known to attacker
Secret msg pad Evil data H(forged)

Real-world exploits: Flickr API (2009), early AWS signature schemes, countless custom MAC constructions using SHA-256(key || msg).

BLAKE3 Uses a finalization flag that seals the output. Structurally immune — no state leaks through the hash.
02

Merkle-Damgard weaknesses

Theoretical

SHA-256's underlying construction has multiple proven theoretical vulnerabilities beyond length extension.

IV f f H
Multicollisions (Joux '04) Herding attacks Long-msg 2nd preimage

Joux showed finding 2k collisions costs only k times more than finding one pair — far less than expected for an ideal hash.

BLAKE3 Built on a Merkle tree, not a Merkle-Damgard chain. Each node is independent — the construction itself eliminates these classes of attacks.
03

Sequential-only processing

Performance

Every block depends on the output of the previous block. One core, always — no matter how many your device has.

SHA-256
1 2 3 4
4 units
BLAKE3
1 2 3 4
1 unit

Your phone has 6 cores. Your laptop has 8-16. SHA-256 uses exactly one of them. The rest sit idle.

BLAKE3 Tree structure means all chunks process simultaneously. More cores = proportionally faster. Linearly.
04

No built-in domain separation

Exploitable

SHA-256 uses the same algorithm regardless of purpose. A hash used as a MAC, KDF, or PRF all look identical — making misuse easy and dangerous.

MAC SHA-256
KDF SHA-256
PRF SHA-256
Same function. No isolation.

Using SHA-256(key || message) as a MAC? Broken by length extension. HMAC-SHA-256 works but is a workaround — two nested hashes because the primitive can't do it alone.

BLAKE3 Built-in keyed hashing, KDF, and PRF modes with domain separation flags. One primitive, safe in every context.
05

Quantum vulnerability

Future risk

Grover's algorithm halves the effective security of any hash function. SHA-256 drops from 256-bit to 128-bit preimage resistance against a quantum computer.

Today
256-bit
Post-quantum
128-bit

128-bit is still strong — but it's half what you thought you were paying for. And quantum computing timelines keep accelerating.

BLAKE3 Also 256-bit (same quantum reduction), but its faster speed means you can affordably use longer outputs when post-quantum margins matter.
06

No verified streaming

Performance

With SHA-256, you must download the entire file before verifying integrity. If it's corrupted or tampered with, you don't know until the end.

SHA-256
chunk chunk chunk verify
BLAKE3
verify verify verify verify

Downloading a 2 GB file over a flaky connection? SHA-256 makes you wait until byte 2,147,483,648 to tell you byte 7 was corrupted.

BLAKE3 Tree structure enables verified streaming — verify each chunk independently as it arrives. Reject bad data immediately.

Open questions

This benchmark is a living experiment. Here's what we're still investigating — and where your device's results might help.

Does multi-core BLAKE3 overtake hardware SHA-256 at 10 MB?

BLAKE3's tree hashing should dominate at large data sizes, but Web Worker overhead is real. We're looking for the crossover point where parallelism definitively wins.

How much would WASM SIMD accelerate BLAKE3?

BLAKE3 is designed to exploit SIMD instructions (NEON on ARM, SSE/AVX on x86). The current WASM implementation doesn't use WASM SIMD — enabling it could significantly widen the gap.

Why do BLAKE3 and SHA-256 WASM tie at 1 KB?

Both show identical throughput at very small sizes. Is this the browser timer resolution floor? Fixed per-call overhead? We're investigating whether this is a measurement artifact or a real performance ceiling.

How do older iPhones compare?

We've tested on recent devices, but iPhone SE, iPhone 11, and other older models have different chips and fewer cores. If you're on an older device, your results are especially valuable.

Should parallel BLAKE3 use full Merkle tree assembly?

Currently, multi-core BLAKE3 hashes independent chunks across workers. A full Merkle tree implementation would combine chunk hashes into a single root — matching the BLAKE3 spec more faithfully and potentially improving large-data throughput.
